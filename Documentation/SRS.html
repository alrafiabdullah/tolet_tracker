Introduction

This section describes the topview of To-Let Tracker which includes both the fuctional and non-fuctional requirements in this SRS document. It has all the explanations so that it could be readable to a vast range of people.

1.1 Document Purpose

The Software Requirement Specification's(SRS) purpose is to present the definitions and describe all the specification of our project To-Let Tracker. This documentation defines the system's primary uses and the most important functionalities so that the next developer who wants to extend it further can easily understand and recognized our work pattern and know where and how to change it if it is required.

1.2 Product Scope

To-Let Tracker is an online platform where people will be able to find their desired flat/apartment to rent easily. What is so special about "To-Let Tracker" you ask? Here, with Augmented Reality users can see the which floors which side flat/apartment is ready for rent. Also, the users will be able to zoom in and out to correctly understand the location of the flat/apartment which will give them more satisfaction.
The owner will have to provide the exact information of his flat/apartment and upload it in our platform. Then the location of the property will be identified by Google Map and the Augmented Reality will show the exact flat/apartment as available via the phone camera. The camera will be able to detect floors. The user will be able to check the flat/apartment from different direction.
The Augmented Reality(AR) feature will be supported by an android app. Also, there will be a web app where both owner and user will be able to find their desired flats/apartments but there will not be any AR option.

1.3 Intended Audience

This SRS is intended for the software developers, designers and testers. It should be read chronologically the way it is written.

Overall Description

This section's purpose is to give overview of the overall system. The system will be explained and show how it is going to interact with other systems. Some of the basic fucntionalities will be described too. At last, the assumptions and dependencies will be described.

2.1 Product Overview

There are two parts in this system: one android app and one web portal. The android application will be used by the users who want to rent flat/apartment and view the exact one in AR. The web portal is going to be used by both property owners and users. The owners will add the information of their properties and the users will be able to check them.
The mobile app needs Android 7.0 or above because the hardwares that will be used for AR can only be found from the Android version 7.0 and above. Also, the mobile phone's camera will be needed to implement the AR features.
The web application and mobile application will both have a database to store the flat/apartment's data and user data. The web portal will have the functionality of reading, writing and modifying access of the database where the mobile app will only have the access to read. The communication with database will happen through the same network for now.

2.2 Product Functionality

The main fucntionality of the mobile app user will be the ability to visualise the exact flat/apartment through AR. The users will also be able to find flats/apartments based on the location they choose.
The web portal will consist of two parts: one for the administrator and the other one is for users. The users will be able to look for flats/apartments from here also. But it won't support AR feature. So, the web portal can be accessed from any device as it won't have any hardware related limitation.

2.3 User Classes and Characteristics

There are two user classes: owners/administrators and users.
The users will use the mobile app and the web app. All the features the mobile app can provide, they will be able to use it.
The administrator will take the information of the flat/apartment and list them on the web portal. The mobile application will not have any management features.

2.4 Operating Environment

Android:

Operating System: Android 7.0
Disk Space: 100 MB
RAM: 1 GB

2.5 Design And Implementation Constraints

Since this is an online application, the first and foremost constraint will be the connectivity. In this case, the local network. It is cruicial for the application to fetch and show the data from our database.
The capacity of the database will be constrained for both mobile and web application. Also, the database is shared between both platforms. So, it may have to force to queue for requests. Therefore, it may take time more than usual to fetch the data.
The android version will need access of multiple sensors like camera, gyro, accelerometer etc. Moreover, it requires android version 7.0 and above to use AR feature.

2.6 Assumptions And Dependencies

One assumption is the phones with good camera will provide better performance than others.
Another assumption is the moderately old released Android phone will meet the hardware requirements for the AR as the current version of Android is 10.
Also, a proper internet browser with stable internet connection is required for the web app.

The software will work only in some specific area where the House Owner or Landlord is searching for renter. The app will show no result when the area have no vacant flat or house.

There will be a dedicated database for User's informations. The informations will be fetched from users through registration process. The password will be recorded using the Hashing function. So the password will be encripted and unaccessable to the naked eye.
